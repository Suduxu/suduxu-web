Suduxu is built around a centralized event system. The native runtime emits structured events that are dispatched into typed channels on the host side (Rust, Unity, or other bindings).

These events are the primary mechanism for reacting to runtime behavior.

---

## Event categories

Suduxu groups events into five core domains:

- TCP events (connection lifecycle, errors)
- UDP events (input, sensor data, screenshots)
- State events (battery, network)
- Health events (device availability)
- Log events (diagnostics and debugging)

---

## Event flow

All events originate from the native layer and are routed through a unified callback system:

```text
Native runtime → JSON event → EventObject → handler dispatch → typed API events
```

Each event is:
- Serialized as JSON in the native layer
- Passed through an FFI callback
- Deserialized into an EventObject
- Routed into a specific handler

## TCP-Events
<Tabs items={["Rust", "Unity"]}>
<Tab value="Rust">
Handled via:
- `ON_TCP_START()`: Emitted when the TCP server starts.
- `ON_TCP_STOP()`: Emitted when the TCP server stops.
- `ON_CLIENT_CONNECTED(id: u16)`: Emitted after a client connects and completes the handshake with optional authentication.
- `ON_CLIENT_DISCONNECTED(id: u16)`: Emitted when a client disconnects.
- `ON_ILLEGAL_SUDUXU_METHOD(payload: Payload)`: Emitted when an illegal method is called.

Example usage:

```rs
ON_CLIENT_CONNECTED.subscribe(|id| {
    println!("Client connected: {}", id);
});
```
> See Reference/Data-Models for details on the [`Payload`](/docs/reference/data-models#payload) data models.

</Tab>
<Tab value="Unity">
Handled via:
- `suduxu.OnTcpStart()`: Emitted when the TCP server starts.
- `suduxu.OnTcpStop()`: Emitted when the TCP server stops.
- `suduxu.OnClientConnected(id)`: Emitted after a client connects and completes the handshake with optional authentication.
- `suduxu.OnClientDisconnected(id)`: Emitted when a client disconnects.
- `suduxu.OnIllegalSuduxuMethod(payload)`: Emitted when an illegal method is called.

Example usage:

```cs
suduxu.Server.OnClientConnected += id =>
{
    Debug.Log($"Client connected: {id}");
};
```
> See Reference/Data-Models for details on the [`Payload`](/docs/reference/data-models#payload) data models.

</Tab>
</Tabs>

## UDP-Events

UDP events cover all real-time device input and sensor data.

Events include:
- Button input
- Joystick input
- Sensor data (accelerometer, gyroscope, etc.)
- Screenshot events

<Tabs items={["Rust", "Unity"]}>
<Tab value="Rust">
Handled via:
- `ON_SENSOR_DATA(sensor_data_raw: SensorDataRaw)`: Emitted when raw sensor data is received.
- `ON_UDP_START()`: Emitted when the UDP subsystem starts.
- `ON_UDP_STOP()`: Emitted when the UDP subsystem stops.
- `ON_BUTTON_INPUT(id: u16, input: ButtonInput)`: Emitted when button input is received from a client.
- `ON_JOYSTICK_DATA(id: u16, input: JoystickData)`: Emitted when joystick data is received from a client.
- `ON_SCREENSHOT(id: u16, path: String)`: Emitted when a screenshot is received from a client.

Example usage:

```rs
ON_SENSOR_DATA.subscribe(|data| {
    println!("Received sensor data: {:?}", data);
});
```

> See Reference/Data-Models for details on the [`SensorDataRaw`](/docs/reference/data-models#sensordataraw), [`ButtonInput`](/docs/reference/data-models#buttoninput) and [`JoystickData`](/docs/reference/data-models#joystickdata) data models.

</Tab>
<Tab value="Unity">
Handled via:
- `suduxu.Input.Broadcast().OnSensorData(sensorDataRaw)`: Emitted when raw sensor data is received.
- `suduxu.Input.Broadcast().OnUdpStart()`: Emitted when the UDP subsystem starts.
- `suduxu.Input.Broadcast().OnUdpStop()`: Emitted when the UDP subsystem stops.
- `suduxu.Input.Broadcast().OnButtonInput(id, input)`: Emitted when button input is received from a client.
- `suduxu.Input.Broadcast().OnJoystickData(id, input)`: Emitted when joystick data is received from a client.
- `suduxu.Input.Broadcast().OnScreenshot(id, path)`: Emitted when a screenshot is received from a client.

Example usage:

```cs
suduxu.Input.Broadcast().OnSensorData += data =>
{
    Debug.Log($"Received sensor data: {data}");
};
```

<Callout type="warning">
Input events that incoorporate the id parameter (including `OnSensorData`) will only trigger for the client matching the id. Use broadcast events for global input handling.
</Callout>

> See Reference/Data-Models for details on the [`SensorDataRaw`](/docs/reference/data-models#sensordataraw), [`ButtonInput`](/docs/reference/data-models#buttoninput) and [`JoystickData`](/docs/reference/data-models#joystickdata) data models.
</Tab>
</Tabs>

## State-Events

State events represent continuous device telemetry.
- Battery changes
- Network changes
- Device availability

<Tabs items={["Rust", "Unity"]}>
<Tab value="Rust">
Handled via:
- `ON_BATTERY_CHANGE(id: u16, battery: Battery)`: Emitted when a client's battery state changes (e.g. battery level or charging status).
- `ON_NETWORK_CHANGE(id: u16, network: Network)`: Emitted when a client's network state changes (e.g. connectivity, type).
- `ON_HEALTHY(id: u16)`: Emitted when a client responds to a health check in time, indicating it is healthy and responsive.
- `ON_UNHEALTHY(id: u16)`: Emitted when a client fails to respond to a health check in time, indicating it may be unresponsive or disconnected.
- `ON_TIMEOUT(id: u16)`: Emitted when a clients responds after the health check timeout has already passed, indicating a potential connectivity issue.

Example usage:

```rs
ON_BATTERY_CHANGE.subscribe(|(id, battery)| {
    println!("Client {} battery changed: {:?}", id, battery);
});
```

> See Reference/Data-Models for details on the [`Battery`](/docs/reference/data-models#battery) and [`Network`](/docs/reference/data-models#network) data models.
</Tab>
<Tab value="Unity">
Handled via:
- `suduxu.Client.Broadcast().OnBatteryChange(id, battery)`: Emitted when a client's battery state changes (e.g. battery level or charging status).
- `suduxu.Client.Broadcast().OnNetworkChange(id, network)`: Emitted when a client's network state changes (e.g. connectivity, type).
- `suduxu.Client.Broadcast().OnHealthy(id)`: Emitted when a client responds to a health check in time, indicating it is healthy and responsive.
- `suduxu.Client.Broadcast().OnUnhealthy(id)`: Emitted when a client fails to respond to a health check in time, indicating it may be unresponsive or disconnected.
- `suduxu.Client.Broadcast().OnTimeout(id)`: Emitted when a clients responds after the health check timeout has already passed, indicating a potential connectivity issue.

Example usage:
```cs
suduxu.Client.Broadcast().OnBatteryChange += (id, battery) =>
{
    Debug.Log($"Client {id} battery changed: {battery}");
};
```

<Callout type="warning">
State events that incorporate the id parameter will only trigger for the client matching the id. Use broadcast events for global state handling.
</Callout>

> See Reference/Data-Models for details on the [`Battery`](/docs/reference/data-models#battery) and [`Network`](/docs/reference/data-models#network) data models.
</Tab>
</Tabs>

## Event rules

- Events are asynchronous
- Delivery order is not guaranteed across categories
- TCP events relate to connection lifecycle only
- UDP events represent real-time input streams
- State events reflect periodic device updates
- Health events may lag behind actual disconnects due to timeout windows

---

## Key constraint

Events are only valid after the runtime has been launched. Subscribing before initialization may result in missing early emissions.

<Callout type="info">
In order to use the `suduxu` object in Unity it first needs to be injected, this is achieved the following way:

```cs
public class SuduxuExample : MonoBehaviour {
  [injected]
  private Suduxu suduxu;
}
```

Otherwise it can also be called like this:
```cs
Suduxu.Instance
```
</Callout>
